## Recent feature additions (the bits a fresh session won't infer)

**Sticky pricing widget** — Steps 1/2/3 of `/book` use a 2-col grid with a sticky right-side `PricingWidget` (desktop) and a fixed-bottom `PricingMobileBar` (mobile). The mobile bar lives OUTSIDE the grid as a sibling — putting both in the same fragment broke desktop stickiness. Files: [app/book/booking-form.tsx](app/book/booking-form.tsx), [app/book/_components/pricing-widget.tsx](app/book/_components/pricing-widget.tsx).

**Optional account creation at checkout** — Step 2 has a "Save your info?" checkbox at the bottom (anonymous bookers only). If ticked + password set (>= 8 chars), the booking-create API provisions a Supabase Auth user via `auth.admin.createUser({ email_confirm: true })`, signs them in on the same request, and stamps `auth_user_id` on the new customer row. Email collisions silently fall back to anonymous booking. Files: [app/book/booking-form.tsx](app/book/booking-form.tsx), [app/api/bookings/create/route.ts](app/api/bookings/create/route.ts), [lib/bookings/schema.ts](lib/bookings/schema.ts), [app/book/confirmed/page.tsx](app/book/confirmed/page.tsx).

**Auto-claim anonymous bookings on sign-in** — Policy reversal from the original "never match by email" stance. Sign-in is now treated as proof of email ownership; orphan customer rows under the user's email get their bookings re-pointed to the user's customer row and the orphans deleted. Idempotent. File: [lib/customers/current.ts](lib/customers/current.ts).

## Tech stack (quick reference)

- Next.js 16 App Router, TypeScript strict, Tailwind v4 (CSS-first `@theme`)
- Supabase: Postgres + Storage + Auth (`@supabase/ssr` for cookie-aware server client)
- Square Web Payments SDK + `react-square-web-payments-sdk` (inline tokenization)
- Dropbox Sign (`@dropbox/sign` + `hellosign-embedded`) — embedded iframe, not redirect
- Resend for transactional email with PDF attachments (idempotent dispatcher in `lib/email/booking-confirmation.ts`)
- Money stored as integer cents
- Webhook-driven status transitions (Square `payment.updated`, Dropbox Sign `signature_request_signed`)

## Conventions

- All status transitions fire from webhooks, never client-side success indicators.
- Webhook handlers read raw body via `request.text()` before verifying signature (never `.json()` first).
- Service-role Supabase client (`lib/supabase/service.ts`) is `import "server-only"` — never importable from a client component.
- `end_date` is the pickup date, not the last day with equipment. Hotel check-in/out semantics. `rentalDays = end - start` (no +1).
- Customer row dedup is by `auth_user_id`, never by email (partial unique index `WHERE auth_user_id IS NOT NULL`).
